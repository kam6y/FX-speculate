# Codex連携

- CC(claude code)は司令塔（設計・計画）、Codexは実行者（実装・修正・テスト生成・レビュー）
- コードを書く・修正・レビューするタスクはcodex MCPツールに自動委譲する
- ただし5行以内の自明な変更はCC自身が直接行ってよい
- codexツール呼び出し時、developer-instructionsにタスク固有のコンテキストを渡す:
  - タスク内容、対象ファイルのパスと役割、設計方針、完了条件
  - コーディング規約はAGENTS.mdでCodex側に設定済みのため不要
- Codexの結果は必ず差分を確認し、品質基準を満たしてからユーザーに報告する
  - 品質基準は別のcodexのレビューで緊急度のhighがない状態です。
  - high以上があった場合、修正後また別のcodexにレビューさせてください。
  - これをhighが消えるまで繰り返します。
- 問題があればcodex-replyで追加指示を出して修正させる

## Codexパラメータ指針

タスク種別ごとに以下の設定を使う:

| タスク | sandbox | approval-policy |
|---|---|---|
| レビュー・調査 | `read-only` | - |
| 実装・修正 | `workspace-write` | `on-failure` |
| テスト実行を伴う実装 | `workspace-write` | `on-failure` |
| 破壊的変更（ファイル削除等） | `danger-full-access` | `untrusted` |

- `developer-instructions`には以下を含める:
  - タスク内容の要約
  - 対象ファイルのパスと役割
  - 設計方針・制約条件
  - 完了条件（何をもって完了とするか）
- `cwd`は常にプロジェクトルートを指定する
- `model`は指定しない（デフォルトを使用）

## CCの役割と作業プロセス

CCは以下のプロセスで作業を進める:

1. **調査**: Glob/Grep/Readで対象コードを把握する
2. **設計**: 変更方針を決定し、必要に応じてユーザーに確認する
3. **委譲**: Codexに具体的な実装タスクを投げる
4. **差分確認**: Codex完了後、`git diff`で変更内容を確認する
5. **レビュー**: 別のCodexセッション（`read-only`）でレビューさせる
6. **報告**: highがない状態になったらユーザーに結果を報告する

差分確認の方法:
- `git diff`で変更ファイル・変更行を確認
- 意図しないファイルが変更されていないか確認
- 変更がタスクのスコープ内に収まっているか確認
